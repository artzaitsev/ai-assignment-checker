## Вопрос 1  таблица clients (0-3)
**Что засчитывается (ключевые идеи, каждая  1 балл):**
1) Не хранить `age` как число (возраст меняется). Правильно: хранить `birth_date` (дата рождения) и возраст вычислять.
2) Ограничения/ключи: `client_id`  PK; `login`  UNIQUE; `city_id`  FK на справочник городов.
3) Контакты: упоминание/добавление полей email/телефон и т.п.

**Шкала:**
- 3: есть все 3 идеи
- 2: есть 2 идеи
- 1: есть 1 идея
- 0: нет содержательной критики

**Примеры фраз (рус):**
- Возраст хранить нельзя, лучше хранить дату рождения.
- Логин должен быть уникальным, city_id  внешний ключ.

## Вопрос 2  историчность цены товара (0-3)
**Что засчитывается:**
1) Нельзя хранить цену только одним полем в `items`, нужна история.
2) Отдельная таблица истории цен (например `item_prices`).
3) Период действия цены: `valid_from/valid_to` (или аналог) и логика новая цена = новая запись (SCD2/temporal).

**Шкала:**
- 3: отдельная таблица + период + логика изменения
- 2: отдельная таблица + есть идея фиксации дат/истории
- 1: только общие слова нужна история, без структуры
- 0: по сути оставляет 1 поле без истории

**Примеры фраз (рус):**
- Сделать таблицу item_prices: item_id, price, valid_from, valid_to.
- При смене цены добавлять новую строку, старую закрывать датой valid_to.

## Вопрос 3  исправление SQL (0-3)
**Ключевые исправления (важные пункты):**
1) `MANAGER_ID = NULL`  должно быть `MANAGER_ID IS NULL`.
2) Условие по сумме зарплат должно быть в `HAVING`, а не в `WHERE`.
3) Более 100000  `SUM(salary) > 100000`.
4) Корректные JOIN и GROUP BY, корректная фильтрация по городу Seattle/Сеул (по условию задания).

**Шкала:**
- 3: IS NULL + HAVING + >100000 + корректная группировка/joins
- 2: исправлено минимум 3 ключевых пункта
- 1: исправлено 12 пункта
- 0: запрос не исправлен по сути

**Примеры фраз (рус):**
- NULL сравнивается через IS NULL.
- SUM() фильтруется через HAVING.
